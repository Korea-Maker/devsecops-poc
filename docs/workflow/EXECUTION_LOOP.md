# EXECUTION_LOOP.md — 솔로 개발자 실행 루프

> 1인 개발 + Claude Code 워크플로우에 최적화된 일일/주간 실행 루프.
> 핵심 원칙: **Explore → Plan → Implement → Verify → Commit**

---

## 일일 실행 루프

### 1. 시작 (15분)

```
[ ] 어제 커밋 로그 확인 (git log --oneline -10)
[ ] 현재 Phase 및 남은 작업 확인 (MASTER_PLAN.md)
[ ] 오늘의 목표 1~2개 설정 (더 이상 잡지 않는다)
[ ] 블로커 확인: 미결정 사항이 오늘 작업을 막는가?
```

### 2. Explore (10~20분)

현재 작업 대상의 코드/문서/의존성을 탐색한다.

```
[ ] 관련 파일 읽기 (코드, 테스트, 문서)
[ ] 의존하는 외부 API/라이브러리 문서 확인
[ ] 기존 패턴 파악 (프로젝트 내 유사 구현이 있는가?)
```

**Claude Code 활용**:
- `/explore` 또는 `Explore` 에이전트로 코드베이스 탐색
- `architect` 에이전트로 설계 방향 검토 (복잡한 경우)

### 3. Plan (10~15분)

구현 전에 반드시 계획을 세운다.

```
[ ] 변경할 파일 목록 정리
[ ] 구현 순서 결정 (의존성 순서)
[ ] 엣지 케이스 미리 식별
[ ] 테스트 전략 결정 (어떤 테스트를 쓸 것인가)
```

**Claude Code 활용**:
- 2개 이상 파일 변경 → `/team` 사용
- 아키텍처 결정 → `/plan` 사용
- 불확실하면 `analyst` 에이전트로 요구사항 명확화

### 4. Implement (2~4시간 × 2블록)

타임박싱된 블록 단위로 구현한다.

```
[ ] 블록 1: 핵심 구현 (2~4시간)
    - 하나의 기능/모듈에 집중
    - 테스트와 함께 작성 (TDD 권장)
    - 중간 커밋 자주 (30분~1시간 단위)
[ ] 휴식 (15~30분)
[ ] 블록 2: 보조 작업 또는 계속 구현 (2~4시간)
    - 테스트 보강, 리팩터링, 문서화
    - 또는 다음 기능 시작
```

**Claude Code 활용**:
- `executor` 에이전트로 구현 위임
- `test-engineer` 에이전트로 테스트 전략/작성
- `build-fixer` 에이전트로 빌드 오류 해결

### 5. Verify (15~30분)

모든 구현 후 반드시 검증한다.

```
[ ] 린트 검사 통과
[ ] 타입 검사 통과
[ ] 테스트 전체 통과
[ ] 보안 스캔 통과 (해당 시)
[ ] 수동 테스트 (핵심 플로우)
[ ] 변경 사항 리뷰 (git diff)
```

**Claude Code 활용**:
- `verifier` 에이전트로 완료 검증
- `security-reviewer` 에이전트로 보안 리뷰 (보안 관련 변경 시)
- `quality-reviewer` 에이전트로 코드 품질 리뷰

### 6. Commit (5~10분)

검증 완료된 변경만 커밋한다.

```
[ ] 커밋 대상 파일 확인 (git status)
[ ] 시크릿/민감 정보 포함 여부 확인
[ ] 커밋 메시지 작성 (type(scope): 설명)
[ ] Co-Authored-By 태그 포함 (Claude 기여 시)
[ ] push (필요 시)
```

### 7. 마무리 (10분)

```
[ ] 오늘 한 일 한 줄 요약 (커밋 로그로 대체 가능)
[ ] 내일 이어갈 작업 메모
[ ] 블로커/미결정 사항 기록
[ ] MASTER_PLAN.md 체크리스트 업데이트
```

## 주간 실행 루프

### 월요일: 주간 계획

```
[ ] 지난주 완료 항목 점검
[ ] 이번 주 목표 3~5개 설정
[ ] 현재 Phase 진행률 확인
[ ] 블로커 해결 계획 수립
[ ] QUESTIONS_FOR_JONGWOOK.md 미답변 항목 확인
```

### 수요일: 중간 점검

```
[ ] 주간 목표 대비 진행률 확인
[ ] 지연 항목 원인 분석
[ ] 남은 3일 우선순위 재조정
[ ] 기술 부채 축적 여부 점검
```

### 금요일: 주간 회고 + 정리

```
[ ] 이번 주 완료 항목 정리
[ ] 잘한 것 / 개선할 것 1줄씩
[ ] 다음 주 핵심 목표 초안
[ ] 코드 정리: 미완성 브랜치, TODO 주석 점검
[ ] MASTER_PLAN.md 의사결정 추적 테이블 업데이트
```

## 타임박싱 규칙

| 활동 | 최대 시간 | 초과 시 행동 |
|------|-----------|-------------|
| Explore | 30분 | 탐색 중단, 현재 이해로 Plan 진입 |
| Plan | 20분 | 계획 확정, Implement 시작 |
| 단일 기능 구현 | 4시간 | 현재까지 커밋, 접근 방식 재평가 |
| 버그 디버깅 | 1시간 | 접근 변경 또는 `debugger` 에이전트 위임 |
| 기술 의사결정 | 30분 | 현재 정보로 결정. 완벽한 선택은 없다 |
| 문서 작성 | 1시간 | 핵심만 기록, 나머지는 TODO |
| 코드 리뷰 | 30분 | 크리티컬만 수정, 나머지 이슈 기록 |

> **원칙**: 타임박스 초과 = 접근 방식이 잘못되었다는 신호. 멈추고 재평가.

## 컨텍스트 윈도우 위생 규칙

Claude Code를 효과적으로 사용하기 위한 규칙:

### 세션 관리

```
1. 하나의 세션 = 하나의 목표
   - 기능 구현과 버그 수정을 같은 세션에서 섞지 않는다
   - 목표 달성 또는 2시간 경과 시 새 세션 시작

2. 세션 시작 시 컨텍스트 설정
   - 현재 작업 목표 명시
   - 관련 파일 경로 제공
   - 제약 조건 명시

3. 세션 종료 시 상태 저장
   - 미완료 작업을 TODO 또는 이슈로 기록
   - 다음 세션에서 이어갈 수 있도록 메모
```

### 프롬프트 위생

```
1. 구체적으로 요청
   - Bad:  "이 코드 고쳐줘"
   - Good: "src/scanner/run.ts의 parseResult 함수에서 null 응답 시 에러 처리 추가"

2. 범위 제한
   - Bad:  "전체 리팩터링 해줘"
   - Good: "src/api/routes.ts에서 중복된 에러 핸들링을 미들웨어로 추출"

3. 검증 기준 명시
   - Bad:  "테스트 작성해줘"
   - Good: "parseResult 함수에 대해 정상/null/에러 케이스 단위 테스트 작성"
```

### 에이전트 활용 패턴

```
상황별 권장 에이전트:

탐색/이해    → explore (haiku) — 가볍고 빠름
설계 검토    → architect (opus) — 깊은 분석
구현         → executor (sonnet) — 표준 구현
복잡한 구현  → deep-executor (opus) — 자율적 실행
디버깅       → debugger (sonnet) — 근본 원인 분석
테스트       → test-engineer (sonnet) — 전략 + 작성
리뷰         → quality-reviewer + security-reviewer
검증         → verifier (sonnet) — 완료 증거 수집
```

## 비상 프로토콜

### 막혔을 때

```
1. 30분 이상 같은 문제 → 접근 방식 변경
2. 1시간 이상 → 문제를 더 작은 단위로 분해
3. 2시간 이상 → 해당 작업 보류, 다른 작업으로 전환
4. 하루 종일 → QUESTIONS_FOR_JONGWOOK.md에 기록, 외부 도움 요청
```

### 스코프 크립 감지

```
경고 신호:
- "이것도 같이 하면 좋겠다" → MASTER_PLAN에 있는가? 없으면 기록만
- "이걸 먼저 리팩터링해야..." → 현재 작업에 필수인가? 아니면 나중에
- PR이 500줄 이상 → 분할 필요
- "거의 다 됐다"가 3번 이상 → 범위 재정의 필요
```
