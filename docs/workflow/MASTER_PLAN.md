# MASTER_PLAN.md — DevSecOps PoC 실행 계획

> 스타트업 개발팀을 위한 DevSecOps 플랫폼. 솔로 개발자(종욱) + Claude Code 워크플로우.
> 이 문서는 Phase 0부터 Phase 9까지의 실행 로드맵이다.

---

## Phase 0: 고객 발견 및 문제 검증

### 목표
타겟 고객(스타트업 개발팀)의 실제 Pain Point를 확인하고, 해결할 가치가 있는 문제인지 검증한다.

### 작업
- [ ] 스타트업 CTO/리드 개발자 5명 이상 인터뷰 섭외
- [ ] 인터뷰 스크립트 작성 (현재 보안 프로세스, 도구, 불만 사항)
- [ ] 인터뷰 실시 및 녹취/메모 정리
- [ ] 경쟁사 분석 (Snyk, SonarQube, GitHub Advanced Security 등)
- [ ] 핵심 가설 정리: "스타트업은 X 때문에 보안을 못 하고 있다"

### 산출물
- `docs/discovery/interview-notes.md` — 인터뷰 요약
- `docs/discovery/competitor-analysis.md` — 경쟁사 비교표
- `docs/discovery/hypothesis.md` — 검증된 가설 + Pivot 여부 판단

### 검증 체크리스트
- [ ] 최소 5명 인터뷰 완료
- [ ] 공통 Pain Point 3개 이상 식별
- [ ] "이 도구가 있으면 쓰겠다"는 긍정 반응 3명 이상
- [ ] 경쟁사 대비 차별점 1개 이상 도출

### 안티패턴 가드
- **빌드 먼저 증후군**: 인터뷰 없이 코드부터 쓰지 않는다
- **확증 편향**: 듣고 싶은 답만 골라 듣지 않는다 — 부정적 피드백도 기록
- **과도한 조사**: 2주 이내로 타임박싱. 완벽한 데이터보다 빠른 검증 우선

## Phase 1: 기술 스택 확정 및 개발 환경 구성

### 목표
Phase 0 결과를 바탕으로 기술 스택을 확정하고, 로컬 + CI 개발 환경을 완성한다.

### 작업
- [ ] `QUESTIONS_FOR_JONGWOOK.md` 답변 완료
- [ ] 백엔드 언어/프레임워크 결정 [의사결정 필요]
- [ ] 프론트엔드 프레임워크 결정 [의사결정 필요]
- [ ] 데이터베이스 선정 [의사결정 필요]
- [ ] 인증 방식 결정 [의사결정 필요]
- [ ] 린터/포매터/테스트 도구 설정
- [ ] CI 파이프라인 기본 구성 (lint → type-check → test → build)
- [ ] Docker 개발 환경 구성
- [ ] CLAUDE.md 플레이스홀더를 실제 명령어로 교체

### 산출물
- 프로젝트 초기 구조 (`src/`, `tests/`, `infra/`)
- `package.json` / `pyproject.toml` / `go.mod` 등 (선택에 따라)
- `.github/workflows/ci.yml` — CI 파이프라인
- `Dockerfile` + `docker-compose.yml`
- CLAUDE.md 업데이트 완료

### 검증 체크리스트
- [ ] `git clone` → 의존성 설치 → 테스트 실행이 한 번에 성공
- [ ] CI 파이프라인이 PR에서 자동 실행됨
- [ ] 린트/타입체크/테스트 명령어가 모두 동작
- [ ] Docker로 로컬 실행 가능

### 안티패턴 가드
- **기술 스택 토끼굴**: 비교만 2주 하지 않는다. 3일 내 결정
- **과도한 설정**: 완벽한 CI보다 동작하는 CI 우선
- **premature optimization**: 성능보다 개발 속도 우선

## Phase 2: 핵심 스캐닝 파이프라인 MVP

### 목표
하나의 보안 스캔 종류(SAST 또는 SCA)가 Git 저장소에 대해 실행되고 결과를 저장하는 최소 파이프라인을 완성한다.

### 작업
- [ ] 스캐닝 엔진 선택 및 통합 방식 결정 [의사결정 필요]
- [ ] Git 저장소 클론/접근 모듈 구현
- [ ] 스캐너 실행 래퍼 구현 (컨테이너 기반 권장)
- [ ] 스캔 결과 파싱 및 정규화 (SARIF 또는 자체 포맷)
- [ ] 결과 저장 (DB 스키마 설계 + 저장 로직)
- [ ] API 엔드포인트: 스캔 트리거 + 결과 조회
- [ ] 기본 에러 핸들링 (타임아웃, 스캔 실패 등)

### 산출물
- `src/scanner/` — 스캐너 래퍼 모듈
- `src/api/` — REST API (스캔 트리거, 결과 조회)
- DB 마이그레이션 파일
- 통합 테스트 (저장소 → 스캔 → 결과 확인)

### 검증 체크리스트
- [ ] 테스트 저장소에 대해 스캔이 성공적으로 완료됨
- [ ] 스캔 결과가 DB에 정상 저장됨
- [ ] API로 결과 조회 가능
- [ ] 스캔 실패 시 적절한 에러 응답 반환
- [ ] 테스트 커버리지 80% 이상 (핵심 모듈)

### 안티패턴 가드
- **기능 과잉**: SAST+SCA+DAST 한 번에 하지 않는다. 하나부터
- **자체 스캐너 개발**: 기존 오픈소스 스캐너를 래핑한다. 바퀴 재발명 금지
- **스키마 과설계**: MVP에 필요한 필드만. 확장은 나중에

## Phase 3: 대시보드 및 리포팅

### 목표
스캔 결과를 시각적으로 확인하고, 팀/경영진용 리포트를 생성할 수 있는 UI를 구현한다.

### 작업
- [ ] 프론트엔드 프로젝트 초기화 [의사결정 필요]
- [ ] 대시보드 레이아웃 설계 (와이어프레임)
- [ ] 취약점 목록 뷰 구현 (필터링, 정렬, 검색)
- [ ] 취약점 상세 뷰 구현 (코드 위치, 심각도, 해결 가이드)
- [ ] 요약 통계 카드 (총 취약점, 심각도별 분포, 추이)
- [ ] 리포트 PDF/HTML 자동 생성 기능
- [ ] 반응형 디자인 (모바일 대응)

### 산출물
- `src/web/` 또는 별도 프론트엔드 프로젝트
- 대시보드 페이지 (메인, 상세, 리포트)
- 리포트 템플릿
- E2E 테스트

### 검증 체크리스트
- [ ] 스캔 결과가 대시보드에 정상 표시됨
- [ ] 필터/정렬이 올바르게 동작
- [ ] 리포트 다운로드가 정상 작동
- [ ] 모바일에서 레이아웃 깨지지 않음
- [ ] Lighthouse 접근성 점수 90+ (목표)

### 안티패턴 가드
- **디자인 과집착**: 기능 동작 우선, 디자인 다듬기는 후순위
- **프론트엔드 프레임워크 삽질**: 선택한 프레임워크의 기본 기능만 사용
- **리포트 포맷 과잉**: PDF 하나로 시작. 다양한 포맷은 고객 요청 시 추가

## Phase 4: CI/CD 연동

### 목표
GitHub Actions (우선) 등 CI/CD 도구에서 보안 스캔을 자동 실행하고 결과를 플랫폼으로 전송하는 연동을 완성한다.

### 작업
- [ ] GitHub App 또는 OAuth App 등록
- [ ] Webhook 수신 모듈 구현 (push, PR 이벤트)
- [ ] GitHub Actions 재사용 워크플로우 작성 (`action.yml`)
- [ ] PR 코멘트로 스캔 결과 요약 게시
- [ ] Check Run API 연동 (PR 상태 표시)
- [ ] 연동 설정 UI (토큰 입력, 저장소 선택)
- [ ] GitLab CI 연동 (후순위)

### 산출물
- GitHub Actions marketplace용 액션
- `src/integrations/github/` — GitHub 연동 모듈
- 연동 설정 페이지
- 연동 가이드 문서

### 검증 체크리스트
- [ ] PR 생성 시 자동으로 스캔 실행됨
- [ ] 스캔 결과가 PR 코멘트로 표시됨
- [ ] Check Run 상태가 올바르게 표시됨
- [ ] 연동 해제 후 재연동이 정상 동작
- [ ] Rate limit 처리 구현됨

### 안티패턴 가드
- **모든 CI 동시 지원**: GitHub Actions 먼저 완성. 나머지는 검증된 후 추가
- **GitHub API 직접 구현**: Octokit 등 공식 SDK 사용
- **토큰 하드코딩**: 시크릿 관리 반드시 적용

## Phase 5: 멀티 테넌시 및 SaaS 인프라

### 목표
여러 팀/조직이 독립적으로 사용할 수 있는 멀티 테넌트 아키텍처와 SaaS 운영 인프라를 구축한다.

### 작업
- [ ] 테넌트 격리 전략 결정 (DB 스키마 분리 vs 행 수준 격리) [의사결정 필요]
- [ ] 인증/인가 시스템 구현 (회원가입, 로그인, 역할 관리)
- [ ] 조직(Organization) 모델 설계 및 구현
- [ ] 테넌트별 데이터 격리 적용
- [ ] 클라우드 인프라 구성 [의사결정 필요: AWS/GCP/Azure]
- [ ] 배포 파이프라인 구성 (staging → production)
- [ ] 모니터링 + 알림 설정 (APM, 로그, 업타임)
- [ ] 백업 및 재해 복구 전략

### 산출물
- `infra/` — IaC 코드 (Terraform/Pulumi)
- 인증/인가 모듈
- 멀티 테넌트 미들웨어
- 배포 파이프라인 (GitHub Actions → Cloud)
- 모니터링 대시보드

### 검증 체크리스트
- [ ] 서로 다른 조직 간 데이터 격리 확인 (A 조직이 B 데이터 접근 불가)
- [ ] 회원가입 → 로그인 → 스캔 실행 전체 플로우 정상
- [ ] staging 환경에서 무중단 배포 성공
- [ ] 모니터링 알림이 정상 발송됨
- [ ] 부하 테스트: 동시 10개 조직 스캔 처리

### 안티패턴 가드
- **인증 직접 구현**: Auth0, Clerk, Supabase Auth 등 검증된 서비스 사용 권장
- **과잉 인프라**: 초기에는 단일 인스턴스로 충분. 수평 확장은 필요 시
- **멀티 테넌시 과설계**: 행 수준 격리로 시작. 스키마 분리는 엔터프라이즈 고객 요청 시

## Phase 6: 베타 고객 온보딩 및 피드백 루프

### 목표
5~10개 스타트업 팀을 베타 사용자로 온보딩하고, 실사용 피드백을 수집하여 제품을 개선한다.

### 작업
- [ ] 베타 신청 랜딩 페이지 제작
- [ ] 온보딩 가이드 문서 작성
- [ ] 베타 사용자 초대 및 설정 지원
- [ ] 피드백 수집 채널 구축 (인앱 피드백, Slack 채널, 정기 미팅)
- [ ] 사용 패턴 분석 (어떤 기능을 쓰는지, 이탈 지점)
- [ ] 피드백 기반 우선순위 재조정
- [ ] 크리티컬 버그/UX 이슈 수정 스프린트

### 산출물
- 랜딩 페이지 (별도 또는 제품 내)
- 온보딩 문서
- 피드백 트래커 (Notion, Linear 등)
- 주간 피드백 요약 보고서

### 검증 체크리스트
- [ ] 최소 5개 팀 온보딩 완료
- [ ] 각 팀이 1회 이상 스캔 실행 성공
- [ ] 피드백 20건 이상 수집
- [ ] NPS 또는 만족도 설문 실시
- [ ] 크리티컬 이슈 0건 (해결 완료)

### 안티패턴 가드
- **피드백 무시**: 모든 피드백을 기록하고 분류한다
- **모든 요청 수용**: "해달라는 것" vs "실제 필요한 것" 구분
- **완벽주의**: 베타는 완벽할 필요 없다. 핵심 플로우만 안정적이면 충분

## Phase 7: 온프레미스 배포 옵션

### 목표
보안 요구사항이 높은 고객이 자체 인프라에 설치할 수 있는 배포 옵션을 제공한다.

### 작업
- [ ] 온프레미스 배포 아키텍처 설계 (Docker Compose / Helm Chart)
- [ ] 설치 스크립트 작성 (원클릭 또는 최소 단계)
- [ ] 환경 설정 외부화 (환경 변수, 설정 파일)
- [ ] 에어갭(air-gap) 환경 지원 여부 결정 [의사결정 필요]
- [ ] 업그레이드/마이그레이션 전략 구현
- [ ] 온프레미스 전용 문서 작성
- [ ] 라이선스 키 / 활성화 메커니즘 구현

### 산출물
- `deploy/on-premise/` — 배포 패키지
- Helm Chart 또는 Docker Compose 파일
- 설치/운영 가이드
- 라이선스 관리 모듈

### 검증 체크리스트
- [ ] 깨끗한 VM에서 설치 스크립트로 30분 내 설치 완료
- [ ] SaaS와 동일한 기능이 온프레미스에서 동작
- [ ] 버전 업그레이드가 데이터 손실 없이 완료
- [ ] 외부 네트워크 없이 핵심 기능 동작 (에어갭 지원 시)
- [ ] 설치 가이드만으로 외부인이 설치 가능

### 안티패턴 가드
- **SaaS와 다른 코드베이스**: 동일 코드베이스에서 배포 방식만 분리
- **온프레미스 우선**: SaaS가 주력. 온프레미스는 엔터프라이즈 옵션
- **지원 부담 과소평가**: 온프레미스는 지원 비용이 높다. FAQ/자동진단 도구 필수

## Phase 8: 유료 전환 모델 및 결제 연동

### 목표
무료 PoC에서 유료 플랜으로 전환하는 모델을 구축하고, 결제 시스템을 연동한다.

### 작업
- [ ] 가격 정책 설계 (플랜별 기능/한도 정의)
- [ ] 무료 → 유료 전환 UX 설계 (업그레이드 유도 포인트)
- [ ] 결제 서비스 연동 [의사결정 필요: Stripe / Paddle / 토스페이먼츠]
- [ ] 구독 관리 (플랜 변경, 취소, 환불)
- [ ] 사용량 추적 및 한도 적용
- [ ] 인보이스/영수증 자동 발행
- [ ] 결제 관련 이메일 알림

### 산출물
- 가격 정책 문서
- `src/billing/` — 결제 모듈
- 플랜 관리 UI
- 결제 관련 이메일 템플릿

### 검증 체크리스트
- [ ] 테스트 모드에서 전체 결제 플로우 성공
- [ ] 플랜 업그레이드/다운그레이드 정상 동작
- [ ] 사용량 한도 초과 시 적절한 안내 표시
- [ ] 구독 취소 후 데이터 보존 정책 동작 확인
- [ ] 국내(원화) + 해외(달러) 결제 모두 테스트

### 안티패턴 가드
- **결제 직접 구현**: Stripe/Paddle 등 검증된 서비스 사용. PCI 컴플라이언스 직접 처리 금지
- **복잡한 가격 정책**: 시작은 2~3개 플랜. 커스텀 플랜은 엔터프라이즈만
- **무료 플랜 과대**: 무료에서 유료로 전환할 이유가 있어야 한다

## Phase 9: GTM 본격화 — 인바운드 + 아웃바운드

### 목표
인바운드(콘텐츠, SEO)와 아웃바운드(직접 영업) 채널을 활성화하여 유료 고객을 확보한다.

### 작업
- [ ] 마케팅 웹사이트 제작 (제품 소개, 가격, 사례)
- [ ] 블로그/콘텐츠 전략 수립 (DevSecOps 가이드, 사례 연구)
- [ ] SEO 최적화 (키워드 리서치, 메타데이터, 구조화 데이터)
- [ ] 소셜 미디어 채널 운영 (LinkedIn, Twitter/X)
- [ ] 아웃바운드 대상 리스트 작성 (스타트업 DB, LinkedIn)
- [ ] 콜드 아웃리치 스크립트 작성
- [ ] 제품 데모 준비 (녹화 + 라이브)
- [ ] Product Hunt / Hacker News 런칭 계획

### 산출물
- 마케팅 웹사이트
- 블로그 글 5편 이상
- 아웃바운드 스크립트 및 이메일 템플릿
- 제품 데모 영상
- GTM 대시보드 (리드 추적)

### 검증 체크리스트
- [ ] 웹사이트 트래픽 추적 설정 완료 (GA, Mixpanel 등)
- [ ] 인바운드 리드 첫 유입 확인
- [ ] 아웃바운드 이메일 응답률 5% 이상
- [ ] 데모 → 트라이얼 전환 3건 이상
- [ ] 첫 유료 고객 확보

### 안티패턴 가드
- **제품 없이 마케팅**: Phase 6(베타)까지 완료된 후 본격 GTM
- **모든 채널 동시**: 인바운드 OR 아웃바운드 하나에 집중 후 확장
- **허영 지표**: 방문자 수보다 전환율(트라이얼 → 유료)에 집중
