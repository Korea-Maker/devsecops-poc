## 1. 제품 (Product)

### Q1. 핵심 스캐닝 범위는 무엇인가?

**왜 필요한가**: 첫 MVP에서 어떤 스캔을 지원하느냐에 따라 스캐너 통합 범위와 개발 기간이 결정된다. 모두 한 번에 하면 출시가 늦어진다.

**선택지**:
- (A) SAST만 (코드 정적 분석) — 가장 빠른 MVP
- (B) SAST + SCA (오픈소스 취약점) — 가장 실용적인 조합
- (C) SAST + SCA + Secret 탐지 — 보안 필수 3종
- (D) 위 전부 + DAST (동적 분석) — 풀스택이지만 오래 걸림

**빠른 추천**: **(B) SAST + SCA**. SAST(Semgrep)와 SCA(Trivy)는 오픈소스 스캐너가 성숙하고, 2주 내 통합 가능. Secret 탐지는 Phase 2 이후 추가.

### Q2. 경쟁사 대비 핵심 차별점은 무엇인가?

**왜 필요한가**: Snyk, SonarQube, GitHub Advanced Security 등 기존 도구 대비 "왜 우리 것을 써야 하는가"에 대한 답이 제품 방향을 결정한다.

**선택지**:
- (A) 가격 — 스타트업이 감당 가능한 가격대
- (B) 간편함 — 5분 내 설정, 복잡한 구성 불필요
- (C) 통합 — 파편화된 도구를 하나로 묶는 것
- (D) 한국 시장 특화 — 국내 컴플라이언스(ISMS 등) 지원

**빠른 추천**: **(B) 간편함**. 스타트업 CTO 인터뷰에서 "설정이 복잡하다"가 가장 빈번한 불만. "GitHub 연동 → 5분 내 첫 스캔"을 핵심 가치로 삼으면 차별화 가능.

> 답변:

### Q3. MVP에서 반드시 포함할 기능과 후순위 기능은?

**왜 필요한가**: MVP 범위가 너무 넓으면 출시가 늦어지고, 너무 좁으면 가치를 보여주기 어렵다.

**선택지**:
- (A) 최소: 스캔 실행 + 결과 조회 (CLI/API)
- (B) 핵심: 스캔 실행 + 결과 대시보드 + CI 연동
- (C) 확장: (B) + 알림 + PDF 리포트
- (D) 풀세트: (C) + 멀티 테넌시 + 결제

**빠른 추천**: **(B) 핵심**. 스캔 + 대시보드 + GitHub Actions 연동이면 PoC 시연과 베타 온보딩 모두 가능. 알림·리포트는 피드백 후 추가.

> 답변:

### Q4. 지원할 프로그래밍 언어 우선순위는?

**왜 필요한가**: 스캐너 룰셋과 파서가 언어별로 다르다. 모든 언어를 동시에 지원하면 품질이 떨어진다.

**선택지**:
- (A) JavaScript/TypeScript 우선 — 스타트업 대다수가 사용
- (B) Python 우선 — AI/ML 스타트업 타겟
- (C) JS/TS + Python — 두 언어로 80% 커버
- (D) 언어 무관 (Semgrep 기본 룰셋 전체 적용)

**빠른 추천**: **(C) JS/TS + Python**. 국내 스타트업 대다수가 이 두 언어 스택 사용. Semgrep 기본 룰셋으로 즉시 지원 가능.

## 2. 사용자 (Users)

### Q5. 초기 타겟 사용자의 팀 규모는?

**왜 필요한가**: 팀 규모에 따라 권한 모델, 멀티 테넌시 복잡도, 가격 정책이 달라진다.

**선택지**:
- (A) 1~5명 (초기 스타트업) — 단순한 권한 모델 가능
- (B) 5~20명 (성장 스타트업) — 팀/역할 구분 필요
- (C) 20~50명 (스케일업) — 부서별 격리, 세부 권한 필요
- (D) 규모 무관 — 유연한 권한 모델 필요

**빠른 추천**: **(A) 1~5명**. PoC 단계에서 권한 모델을 복잡하게 만들면 개발이 늦어진다. 단일 조직 + admin/member 2단계면 충분. 성장 후 역할 확장.

> 답변:

### Q6. 주 사용자의 역할은?

**왜 필요한가**: 사용자 역할에 따라 UI/UX 설계, 정보 깊이, 알림 수준이 달라진다.

**선택지**:
- (A) 개발자 — 코드 수준 취약점 상세 정보 필요
- (B) CTO/리드 — 전체 보안 현황 요약 + 추이 필요
- (C) 보안 담당자 — 컴플라이언스 리포트 + 정책 관리 필요
- (D) 복수 역할 — 역할별 다른 뷰 제공

**빠른 추천**: **(A) 개발자**. 스타트업에서 보안 전담자는 드물다. 개발자가 직접 취약점을 확인하고 수정하는 워크플로우에 집중. CTO 뷰는 대시보드 요약으로 대체.

> 답변:

### Q7. 사용자 인증/온보딩 방식은?

**왜 필요한가**: 인증 방식이 온보딩 경험과 개발 복잡도를 크게 좌우한다.

**선택지**:
- (A) 이메일/비밀번호 — 가장 기본
- (B) GitHub/GitLab OAuth — 개발자 친화적, 저장소 연동 자연스러움
- (C) Google Workspace SSO — 기업 환경 친화적
- (D) (B) + (C) 복합

**빠른 추천**: **(B) GitHub OAuth**. 저장소 목록 자동 불러오기 + 인증을 한 번에 해결. 개발자에게 가장 자연스러운 온보딩 경험.

> 답변:

### Q8. 사용자 피드백 수집 방식은?

**왜 필요한가**: 베타 사용자의 피드백을 체계적으로 수집해야 제품을 올바른 방향으로 개선할 수 있다.

**선택지**:
- (A) 인앱 피드백 위젯 (Canny, Hotjar 등)
- (B) 정기 사용자 인터뷰 (주 1회)
- (C) GitHub Issues로 공개 수집
- (D) (A) + (B) 병행

**빠른 추천**: **(C) GitHub Issues**. 추가 도구 비용 0원. 개발자 사용자에게 익숙한 채널. 초기 5~10명 베타에서는 Slack DM + Issues 조합이면 충분.

## 3. 기술 스택 (Tech Stack)

### Q9. 백엔드 언어/프레임워크는?

**왜 필요한가**: 전체 개발 속도, 에코시스템, 채용 가능성, 성능 특성을 결정하는 가장 큰 선택이다.

**선택지**:
| 선택 | 장점 | 단점 |
|------|------|------|
| (A) TypeScript + NestJS/Fastify | 풀스택 통일, 빠른 프로토타이핑 | CPU 집약 작업 성능 한계 |
| (B) Python + FastAPI | AI/ML 에코시스템, 보안 도구 통합 용이 | 타입 안전성 약함 |
| (C) Go + Gin/Echo | 높은 성능, 바이너리 배포 | 에코시스템 작음 |
| (D) Rust + Axum | 최고 성능, 메모리 안전 | 개발 속도 느림 |

**빠른 추천**: **(A) TypeScript + NestJS**. 프론트엔드와 언어 통일, 보안 스캐너는 대부분 CLI 호출이라 성능 병목 없음. NestJS의 모듈 구조가 확장에 유리.

> 답변:

### Q10. 프론트엔드 프레임워크는?

**왜 필요한가**: 대시보드 UX 품질과 개발 속도를 결정한다.

**선택지**:
| 선택 | 장점 | 단점 |
|------|------|------|
| (A) Next.js (React) | SSR/SSG, 거대한 에코시스템 | 번들 크기, 복잡성 |
| (B) Nuxt (Vue) | 직관적 API, 좋은 DX | React 대비 생태계 작음 |
| (C) SvelteKit | 뛰어난 성능, 적은 보일러플레이트 | 작은 커뮤니티 |
| (D) HTMX + 서버 템플릿 | 단순함, JS 최소화 | 복잡한 인터랙션 어려움 |

**빠른 추천**: **(A) Next.js**. shadcn/ui + Tailwind로 대시보드를 빠르게 구축 가능. Vercel 무료 배포로 인프라 비용 0원 시작.

> 답변:

### Q11. 데이터베이스는?

**왜 필요한가**: 스캔 결과의 구조, 쿼리 패턴, 확장성을 결정한다.

**선택지**:
| 선택 | 장점 | 단점 |
|------|------|------|
| (A) PostgreSQL | 범용, JSON 지원, 확장성 | 운영 복잡도 |
| (B) PostgreSQL + Redis (캐시) | 위 장점 + 성능 | 인프라 복잡도 증가 |
| (C) SQLite → PostgreSQL 마이그레이션 | 최소 인프라, 빠른 시작 | 마이그레이션 비용 |
| (D) MongoDB | 스키마 유연성, 문서 기반 | 트랜잭션 약함 |

**빠른 추천**: **(A) PostgreSQL**. Supabase/Neon 무료 티어로 관리형 DB 즉시 사용 가능. SARIF JSON 데이터는 `jsonb` 컬럼으로 처리.

> 답변:

### Q12. ORM/쿼리 빌더는?

**왜 필요한가**: 데이터 접근 패턴의 타입 안전성과 생산성을 결정한다.

**선택지** (백엔드 선택에 따라):
- (A) Prisma — 스키마 기반, 강력한 타입, 마이그레이션 내장
- (B) Drizzle — 경량, SQL에 가까움, 타입 안전
- (C) TypeORM — 데코레이터 기반, NestJS 통합 자연스러움
- (D) Kysely — 타입 안전한 쿼리 빌더, ORM 없이

**빠른 추천**: **(A) Prisma**. 스키마 선언 → 자동 마이그레이션 + 타입 생성. 1인 개발에서 생산성이 가장 높음.

> 답변:

### Q13. 모노레포 vs 멀티레포 전략은?

**왜 필요한가**: 프로젝트 구조가 CI/CD 파이프라인, 의존성 관리, 배포 단위를 결정한다.

**선택지**:
- (A) 모노레포 (Turborepo/Nx) — 코드 공유 쉬움, 단일 PR
- (B) 멀티레포 (백엔드/프론트엔드 분리) — 독립 배포, 명확한 경계
- (C) 단일 레포 (모노레포 도구 없이) — 가장 단순

**빠른 추천**: **(C) 단일 레포**. 1인 개발에 모노레포 도구는 오버엔지니어링. 하나의 레포에 `apps/api`, `apps/web` 디렉터리로 구분하면 충분. 팀이 커지면 Turborepo 도입.

> 답변:

## 4. 데이터 (Data)

### Q14. 스캔 결과 보존 기간은?

**왜 필요한가**: 저장 비용과 컴플라이언스 요구사항에 직접 영향. 오래 보관할수록 비용 증가.

**선택지**:
- (A) 30일 (무료) / 1년 (유료) — 비용 효율적
- (B) 무제한 보관 — 단순하지만 비용 증가
- (C) 사용자 설정 가능 — 유연하지만 구현 복잡

**빠른 추천**: **(A) 30일/1년**. PoC에서는 30일이면 충분. 보존 정책은 DB에 `expires_at` 컬럼 하나 추가로 구현 가능.

> 답변:

### Q15. 스캔 결과 포맷은?

**왜 필요한가**: 외부 도구 연동 및 데이터 이동성에 영향.

**선택지**:
- (A) SARIF (Static Analysis Results Interchange Format) — 업계 표준
- (B) 자체 포맷 — 유연하지만 호환성 낮음
- (C) SARIF 저장 + 자체 정규화 뷰 — 둘 다 지원

**빠른 추천**: **(C) SARIF + 자체 뷰**. Semgrep/Trivy 모두 SARIF 출력 지원. 원본은 SARIF로 저장하고, 대시보드 표시용 정규화 테이블을 별도로 구성.

> 답변:

### Q16. 멀티 테넌시 데이터 격리 전략은?

**왜 필요한가**: 보안 격리 수준과 운영 복잡도의 트레이드오프.

**선택지**:
- (A) 행 수준 격리 (tenant_id 컬럼) — 단순, 비용 효율적
- (B) 스키마 수준 격리 (테넌트별 스키마) — 더 강한 격리
- (C) DB 수준 격리 (테넌트별 DB) — 최강 격리, 높은 비용
- (D) (A)로 시작 → 엔터프라이즈 고객에 (B)/(C) 제공

**빠른 추천**: **(A) 행 수준 격리**. 모든 테이블에 `org_id` 컬럼 추가 + RLS(Row Level Security) 정책. PoC에는 이것으로 충분하고, PostgreSQL RLS가 기본 지원.

> 답변:

### Q17. 스캔 작업 큐 처리 방식은?

**왜 필요한가**: 스캔은 수 초~수 분 소요되는 비동기 작업이다. 요청-응답 방식으로 처리하면 타임아웃이 발생한다.

**선택지**:
- (A) BullMQ (Redis 기반) — Node.js 생태계, 재시도/우선순위 지원
- (B) PostgreSQL LISTEN/NOTIFY + pg-boss — DB만으로 큐 처리
- (C) AWS SQS / GCP Pub/Sub — 클라우드 관리형
- (D) 동기 처리 (소규모 PoC에서)

**빠른 추천**: **(B) pg-boss**. 별도 Redis 없이 PostgreSQL만으로 작업 큐 구현. PoC 규모에서는 성능 충분하고 인프라 단순화.

## 5. 보안 (Security)

### Q18. 플랫폼 자체의 보안 기준은?

**왜 필요한가**: 보안 제품이 보안에 취약하면 신뢰를 잃는다. 최소 기준을 먼저 정해야 한다.

**선택지**:
- (A) OWASP Top 10 대응 — 최소 기준
- (B) SOC 2 Type II 준비 — SaaS 고객 신뢰 확보
- (C) ISMS 인증 준비 — 국내 대기업/공공 고객 대상
- (D) (A)부터 시작, 고객 요구에 따라 상향

**빠른 추천**: **(D) OWASP부터 시작**. PoC 단계에서 SOC 2나 ISMS는 시기상조. OWASP Top 10 체크리스트 대응 후, 베타 고객이 인증을 요구하면 그때 준비.

> 답변:

### Q19. 고객 소스코드 접근 방식은?

**왜 필요한가**: 가장 민감한 보안 결정. 고객 코드를 어떻게 다루느냐가 신뢰의 핵심.

**선택지**:
- (A) 소스코드 서버 전송 후 스캔 — 구현 간단, 보안 우려 큼
- (B) 고객 환경에서 스캔 → 결과만 전송 — 보안 강함, 구현 복잡
- (C) CI 파이프라인에서 스캔 (코드 미전송) — 절충안
- (D) (A) + 스캔 후 즉시 삭제 + 암호화 — 절충안2

**빠른 추천**: **(C) CI 파이프라인 스캔**. GitHub Actions에서 스캐너 실행 → 결과(SARIF)만 API로 전송. 고객 코드가 우리 서버에 오지 않아 신뢰 확보 + 구현도 비교적 단순.
